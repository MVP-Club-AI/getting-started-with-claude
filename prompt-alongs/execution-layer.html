The Execution Layer

Turn your data layer into a system that runs — reliably, repeatedly, and better every time.

Layer 1 — Data & Context Built
Your files, your knowledge, your references. You built this in the previous tutorial.

You have a data layer. Your files are mapped, categorized, and navigable. Claude can find what it needs. That's the foundation — and it's necessary, but it's not sufficient. A database without instructions is just a library with no librarian.

The execution layer is where you tell Claude what to do with all that data. It's the instructions, the processes, the playbooks, the scripts. It's also where Claude starts managing its own state — creating files for itself to track progress, remember decisions, and improve over time.

Writing instructions for a human and writing instructions for Claude are fundamentally different skills.

When you brief a colleague, they fill in gaps with judgment and a Slack message. Claude follows what you wrote literally. Every ambiguity becomes a guess. Learning to see your own ambiguity is the skill this tutorial teaches.

What you'll need
The data layer folder you built in the previous tutorial — with your directory map and context index. Claude Code or Claude Cowork pointed at that folder. A recurring task you want to systematize — something you do weekly or monthly that would benefit from consistent execution.

We're going to start by doing something deliberately imperfect. You're going to write instructions for a task — in your own words, the way you'd explain it to a colleague — and then hand it to Claude with your data layer. We want to see what happens when the instructions are natural language, not engineered.

Prompt #1 — The naive run
I want you to do a task for me. Here are my instructions: [WRITE YOUR INSTRUCTIONS IN PLAIN LANGUAGE — describe the task the way you'd explain it to a colleague over coffee. Don't overthink it. Just tell Claude what you need done, what the output should look like, and any important details.] Use the files in this folder as your reference — check the _context-index.md to understand what's available. Work from my actual data, not generic knowledge. When you're done, add a section at the end called "Where I had to guess." List every place where my instructions were unclear, ambiguous, or missing information — and tell me what you decided to do instead. Be honest and specific.
The "where I had to guess" section is the whole point
The output matters less than the diagnostic. Every place Claude had to guess is a place where your instructions were ambiguous. Those guesses are where output quality degrades — where you get generic filler instead of precisely what you wanted. The list of guesses is your improvement roadmap.

Read the output. Then read the "where I had to guess" section carefully. You'll probably be surprised by how many assumptions Claude had to make. Things that felt obvious to you — because you carry context in your head — weren't in the instructions at all.

This isn't a failure — it's a baseline
The naive run is supposed to be imperfect. It shows you exactly where the gap is between "how you think about the task" and "what you actually wrote down." That gap is the execution layer work. Closing it is the skill you're building.

You have a list of places Claude had to guess. Now let's turn Claude into your collaborator for improving the instructions. This is a conversation — you're working with Claude to make the execution layer stronger.

Prompt #2 — The diagnostic conversation
Let's improve those instructions together. For each place you had to guess: 1. **What was ambiguous?** Quote the specific part of my instructions that was unclear. 2. **What did you assume?** Tell me what you decided and why. 3. **What would have helped?** What could I have written instead that would have removed the ambiguity? Then look at the instructions as a whole and tell me: - What's missing entirely? What did I forget to mention that you needed to know? - What's overspecified? Where did I give you detail that didn't matter? - Where did you lean on the data layer vs. the instructions? Which parts of the task were driven by the reference files, and which by my written instructions? Finally: rewrite the instructions for me. Take everything we just discussed and produce a version that's precise enough you wouldn't need to guess at all. Keep it concise — remove anything that doesn't reduce ambiguity.

The best way to improve your instructions is to ask the system that follows them.

Claude knows exactly where it struggled. It knows what was clear, what was vague, and what was missing entirely. Use that knowledge. This is a collaborative relationship — you manage the instructions, Claude helps you refine them.

Read the rewritten instructions Claude produced. Compare them to your original. Notice what changed. The differences are a map of your implicit assumptions — the things you knew but didn't say, because they felt obvious. In the execution layer, nothing is obvious. Everything has to be explicit.

Check the data layer too
Some of the gaps won't be instruction problems — they'll be data layer problems. If Claude couldn't find a brand guide, that's not an instruction issue. That's a missing file. If Claude didn't know your client's preferences, that's institutional knowledge that needs to be written down. When Claude flags something as missing, ask yourself: is this an instruction I need to write, or context I need to add to my data layer?

You have improved instructions. Let's see if they actually work better. Same task, same data layer, new instructions.

Prompt #3 — The improved run
Let's run the task again. Use the improved instructions we just created — not my original ones. Same rules as before: use the data layer files via the _context-index.md. Don't ask me clarifying questions — work from the instructions and the data. When you're done, add the "Where I had to guess" section again. I want to see if the list got shorter. Also add a section called "What worked well" — tell me which parts of the improved instructions made the biggest difference in output quality.
The "guesses" list should be shorter
If it's not — if Claude is still guessing in the same places — that's a signal that the instructions need a different approach, not just more detail. Maybe the task needs to be broken into smaller steps. Maybe the data layer is missing something fundamental. Either way, the diagnostic tells you where to look.

Compare the two outputs side by side. First run vs. second run. You should see the improved instructions producing output that's closer to what you actually wanted — less generic, fewer wrong assumptions, more aligned with how you'd have done it yourself.

This is the iteration loop that makes the execution layer better over time: run, diagnose, improve, run again. Every cycle tightens the instructions. Every cycle surfaces assumptions you didn't know you were making.

You've seen the iteration loop work. Now let's zoom out. You've been writing instructions in a single conversation. But the execution layer isn't a conversation — it's infrastructure. It's persistent files, reusable commands, and state management that lives in your folder and survives between sessions.

Both Claude Code and Claude Cowork have built-in structures designed exactly for this. Most people don't know they exist — or treat them as one-time setup instead of living documents they iterate on. These structures are the scaffolding of your execution layer.

The execution layer has four components.

Persistent instructions. Task-specific playbooks. Reusable commands. State management. Together, they turn a folder of data into a system that runs.

Execution layer components
Persistent instructions
General rules Claude follows every time. Style, tone, conventions, how you work together.
Task playbooks
Step-by-step instructions for specific recurring tasks, wired into your data layer.
Reusable commands
Named actions you can invoke with a slash. Define once, run anytime.
State management
Files Claude creates for itself — progress logs, decision records, checklists — to manage work over time.

Notice what's happening to your directory. It started as files you created for your own work. Then you added navigation (directory map, context index) so Claude could find things. Now you're adding instructions and state management — files that Claude both reads and writes. The folder is evolving from a filing cabinet into a collaborative workspace where you and Claude both operate.

Time to set up your execution infrastructure. The concepts are the same regardless of which tool you use — persistent instructions, task playbooks, reusable commands, state management. The implementation is different. Pick your path.

1. Persistent instructions — CLAUDE.md

CLAUDE.md is a file that Claude reads automatically every time it starts a session in your project. It's where your general operating instructions live — how you want Claude to work, what conventions to follow, what to always do and never do.

Prompt #4a — Set up CLAUDE.md
Based on everything we've learned working together — the task we ran, the instructions we refined, the data layer in this folder — I want you to help me create a CLAUDE.md file for this project. It should include: 1. **Project context** — What this folder is for, what kind of work it supports. Reference the _context-index.md. 2. **Working conventions** — How I want you to approach tasks. Tone, format, level of detail. Pull this from what we've discovered works well. 3. **Key reference files** — Which files in the data layer to always check before starting a task. Use @imports to point to them (e.g., @reference/brand-guide.md). 4. **Guardrails** — What you should never do. Common mistakes to avoid. Things that are important to me that might not be obvious. 5. **State management rules** — When you create or update files to track progress, where should they go? What format? Keep it concise — aim for 50-100 lines. Every line should reduce ambiguity or prevent a mistake. If removing a line wouldn't change your behavior, cut it.
Where CLAUDE.md lives
Put it in the root of your project folder — that's the default location Claude Code reads from. You can also use .claude/CLAUDE.md if you prefer. For personal preferences that apply across all your projects, use ~/.claude/CLAUDE.md. For modular rules, use .claude/rules/*.md — each file is auto-loaded.
2. Task playbooks — Skills

Skills are reusable instruction sets that live in your project. They can be invoked with a slash command (like /weekly-report) or Claude can activate them automatically when it detects a relevant task.

Prompt #5a — Create your first skill
Take the refined instructions we created for [YOUR TASK] and turn them into a Claude Code skill. Create a skill directory at .claude/skills/[task-name]/ with a SKILL.md file that includes: 1. YAML frontmatter with: name, description (when Claude should use this), and argument-hint if the task takes input. 2. The full execution instructions we refined — step-by-step, with references to data layer files. 3. Quality standards and guardrails. Also ask me: are there any other recurring tasks I do that should become skills? List 3-4 candidates based on what you've seen in my data layer.
Skills compound
Every skill you create is a task you never have to explain again. Instead of pasting instructions each time, you type /weekly-report and Claude has the full playbook — wired into your data layer, with quality standards and guardrails built in. Over time, your .claude/skills/ folder becomes a library of everything you know how to do, encoded as instructions Claude can follow.
3. State management

Claude Code has built-in memory that persists across sessions — auto-memory files in ~/.claude/projects/. But you can also have Claude create project-specific state files in your folder: progress logs, decision records, checklists, run histories. These let Claude manage work over time, not just execute in the moment.

Ask Claude to help you improve
This is a habit, not a one-time setup. After every task run, ask Claude: "How can we improve the instructions? Where did you have to guess? What would make the next run more reliable?" Claude will suggest updates to your CLAUDE.md, your skills, and your data layer. Your execution layer gets better every time you use it.
1. Persistent instructions — Folder instructions

Cowork lets you set persistent instructions for each folder you connect. These instructions apply to every session in that folder — like CLAUDE.md in Claude Code, but configured through the GUI. You can also set global instructions in Settings > Cowork that apply everywhere.

Prompt #4b — Set up folder instructions
Based on everything we've learned working together — the task we ran, the instructions we refined, the data layer in this folder — I want you to help me write the folder instructions for this Cowork project. Write instructions I can paste into the Cowork folder settings that include: 1. **Project context** — What this folder is for, what kind of work it supports. Tell Claude to always check _context-index.md first. 2. **Working conventions** — How I want you to approach tasks. Tone, format, level of detail. 3. **Key reference files** — Which files to always read before starting any task in this folder. 4. **Guardrails** — What to never do. Common mistakes to avoid. 5. **State management rules** — When you create files to track progress or decisions, where should they go? What format? Keep it concise. Every line should reduce ambiguity or prevent a mistake.
Where to set folder instructions
In Claude Cowork, click the folder name in the sidebar, then look for the instructions text area. Paste your instructions there. They'll persist across every session that uses this folder. For instructions that apply to all your Cowork sessions regardless of folder, go to Settings > Cowork.
2. Task playbooks — Plugins & commands

Cowork uses plugins to package reusable workflows. Plugins contain skills (domain knowledge Claude activates automatically), commands (slash-invocable actions), and connectors (integrations with external tools). You can use Anthropic's official plugins or build your own.

Prompt #5b — Create a task playbook
Take the refined instructions we created for [YOUR TASK] and help me turn them into a reusable playbook. Create a file called _playbook-[task-name].md in this folder that includes: 1. **Task overview** — What this produces and who it's for. 2. **Step-by-step execution instructions** — The refined version we built, with explicit references to data layer files. 3. **Quality standards** — What good output looks like. 4. **Guardrails** — What to avoid. Then help me write a Cowork plugin command for this task. Create the directory structure: - .claude-plugin/plugin.json (the manifest) - commands/[task-name].md (the slash command) - skills/[task-name]-knowledge.md (domain knowledge Claude should activate automatically) Also ask me: what other recurring tasks should become commands? List 3-4 candidates based on what you've seen in my data layer.
Commands compound
Every command you create is a task you never have to explain again. Instead of describing what you need each time, you type /weekly-report and Claude has the full playbook — wired into your data layer, with quality standards built in. Over time, your commands become a library of everything you know how to do, encoded as instructions Claude can follow.
3. State management

Cowork starts fresh each session — it doesn't have Claude Code's automatic cross-session memory. But you can work around this by having Claude create state files in your folder: progress logs, decision records, session summaries. At the start of each session, tell Claude to read those files to pick up where you left off.

Ask Claude to help you improve
This is a habit, not a one-time setup. After every task run, ask Claude: "How can we improve the instructions? Where did you have to guess? What would make the next run more reliable?" Claude will suggest updates to your folder instructions, your playbooks, and your data layer. Your execution layer gets better every time you use it.

You have persistent instructions, a task playbook or skill, and a sense of how state management works. Now let's cement the most important habit: the improvement conversation. This is the ongoing practice that turns a static set of instructions into a system that gets better with every run.

Prompt #6 — The improvement conversation
I want to make our working system better. Look at everything we've built — the data layer, the instructions, the task we've run — and help me improve. Answer these questions: 1. **Ambiguity audit.** Where in our instructions could two different AI sessions interpret things differently? Where is there room for inconsistent output? 2. **Data layer gaps.** What reference material, examples, or institutional knowledge is still missing from the data layer that would improve execution quality? 3. **Reliability check.** If I ran this task 10 times in a row, what would vary between runs? What would I need to add to make the output more consistent? 4. **State and memory.** What should Claude be tracking across runs to get smarter over time? What decisions, outcomes, or feedback should be recorded? 5. **New capabilities.** What scripts, tools, or integrations would make this system more powerful? What's a manual step that could be automated? For each answer, give me the specific change — not just "improve the instructions" but exactly what to add, edit, or create.
This conversation is the highest-leverage thing you can do
One improvement conversation — 10 minutes — can save hours of rework across future runs. Every ambiguity you eliminate, every gap you fill, every inconsistency you fix compounds. The system doesn't just get better at this task. The skills you build — seeing ambiguity, thinking about reliability, structuring instructions — transfer to every task you systematize.

Implement the changes Claude suggests. Update your CLAUDE.md or folder instructions. Refine the skill or playbook. Add missing files to the data layer. Then run the task again and see the difference.

After every run, at least briefly. A quick "Where did you have to guess?" after each task takes 30 seconds and catches problems early. A full improvement conversation — like the prompt above — is worth doing after every 3-5 runs, or whenever you notice output quality slipping.

Over time, the frequency drops. The first few runs of any task surface the biggest gaps. After 5-10 iterations, the instructions stabilize and you'll spend less time improving and more time just running.

"Fine" is the enemy of great in execution layer work. If the output is acceptable but not quite what you'd produce yourself, there's a gap in the instructions. The question to ask is: "What do I know about what 'great' looks like that isn't in the instructions yet?" Often the answer is a quality standard that's implicit — you know it when you see it but haven't written it down. Writing it down is the work.

You've built the execution layer for one task. Now let's think about your work as a whole. Every recurring task you do is a candidate for this same treatment: naive run, diagnostic conversation, iteration, persistent instructions, reusable command.

Map your recurring tasks

Think about your typical work week. Which tasks repeat?

  1. What do you do every week that follows roughly the same pattern?
  2. What do you do every month that requires gathering information and producing an output?
  3. What tasks do you dread because they're tedious but important?
  4. What tasks take you disproportionately long because you're always starting from scratch?

Each of these is a candidate for the execution layer. Start with the one that's most painful or most frequent.

One task at a time
Don't try to systematize everything at once. Pick the next most painful recurring task, run it through the same process (naive run → diagnostic → iterate → persist), and add it to your skills or commands. Each one takes less time than the last because you're getting better at the core skill: writing precise instructions and seeing your own ambiguity.

As your execution layer grows, your folder starts to look different. It's not just data anymore — it's a system. Data files, navigation files, instruction files, state files, skills, commands. Two of the three layers, living side by side, with Claude operating on both.

Ask Claude: how can we make this system better?
Make this a regular practice. Not just "how do we improve this task" but "how do we improve the system?" Are there patterns across your skills that should be pulled into CLAUDE.md? Are there data layer files that multiple tasks reference that should be richer? Is there state that would be useful across tasks, not just within one? Claude can see your whole folder. Use that perspective.

You're not writing prompts anymore. You're managing a system.

Persistent instructions. Task playbooks. Reusable commands. State management. An improvement loop that makes everything better with every run. That's the execution layer.
