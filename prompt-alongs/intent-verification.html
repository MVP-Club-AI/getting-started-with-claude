Most people build first, then evaluate. They look at Claude's output and ask themselves: "Is this good?" That question is too late. By the time you're evaluating, you're comparing against a feeling, not a standard. And feelings shift with every read.

The best practitioners do the opposite. They define what "good" looks like before they run anything. They write down their success criteria, build concrete test cases, and then measure output against those standards. The evaluation isn't subjective — it's structured.

If you can't describe what good looks like, you can't tell Claude what to build.

Intent isn't a prompt. It's the answer to "what is this for?" and "how will I know it worked?" Most prompts skip both questions. This tutorial teaches you to answer them first.

What you'll need
The data layer folder and execution infrastructure you built in the data layer and execution layer tutorials. Claude Code or Claude Cowork pointed at that folder. A task you've run at least once — something with output you can evaluate.

Intent is not a prompt. A prompt says "do this thing." Intent answers two deeper questions: What is this for? and How will I know it worked?

Most prompts describe what to build but skip why and what success looks like. The result is output that's technically competent but misses the point. It looks right but doesn't accomplish anything.

Without intent

  • "Write a blog post about AI"
  • "Create a weekly report"
  • "Draft an email to the team"
  • "Summarize this document"

With intent

  • "Write a post that convinces skeptical managers to try AI tools. Success = they forward it to their team."
  • "Create a report that surfaces the 3 decisions my boss needs to make this week. Success = she acts on at least one."
  • "Draft an email that gets the team aligned on the new timeline. Success = zero follow-up questions."
  • "Summarize so my VP can make a go/no-go decision in 2 minutes."
Intent is the difference between "make it good" and "make it persuasive to skeptical readers"
When you define intent, you give Claude a target to aim at — not just a task to complete. The output changes because the goal is different. A blog post meant to persuade is written differently than one meant to inform. A summary for a VP is structured differently than one for an engineer. Intent shapes every decision Claude makes during execution.

This seems simple. It's not. Most of us carry intent in our heads and never write it down. We know what we want — we can feel when the output misses it — but we haven't articulated it in a way that guides execution. That tacit knowledge is the gap this tutorial closes.

The gap between "I'll know it when I see it" and "here's what good looks like" is the entire verification layer.

Crossing that gap is how you stop relying on feel and start building systems that reliably produce what you actually want.

Before you can write specific test cases, you need to define what quality means for your work. Different domains have different dimensions: a marketing team cares about brand voice and persuasion. An analyst cares about accuracy and completeness. A designer cares about consistency and clarity.

A quality standard is a document that defines your dimensions of quality — the specific things that matter for the kind of output you produce. It's not generic ("be professional") — it's specific to your work and your standards.

Prompt #1 — Define your quality standard
I want to define a quality standard for a recurring task I do: [DESCRIBE YOUR TASK — e.g., "writing weekly client updates," "creating project proposals," "drafting social media content"]. Help me create a _quality-standard.md file that defines what "good" looks like for this task. For each quality dimension: 1. **Name the dimension** — e.g., Accuracy, Tone, Completeness, Format, Audience Fit, Persuasiveness, Actionability 2. **Define what it means** — specifically for my work, not generically 3. **Give a "what good looks like" example** — a concrete description of what quality output does in this dimension 4. **Give a "what bad looks like" example** — so the boundary is clear Aim for 5-8 dimensions. Don't include dimensions that don't actually matter for this task — every dimension should be something that, if it failed, would make the output noticeably worse. Save this as _quality-standard.md in the current folder.
This document becomes a reusable asset
Your quality standard isn't just for this one task. It encodes what you care about — your professional judgment crystallized into a document. As you build more tasks and more execution infrastructure, this standard becomes a shared reference across all of them. Claude reads it and knows what "good" means to you.

Read the quality standard Claude produces. Edit it. This is your judgment, not Claude's — Claude helped you articulate it, but the standards should reflect what you actually care about. Remove dimensions that don't matter. Add ones Claude missed. Sharpen the examples until they feel right.

Quality standards define dimensions. Scenarios define specific test cases — concrete situations with observable outcomes. A scenario says: "Given this input, the output should satisfy these specific criteria." It's not a vibe check. It's a structured evaluation.

The key principle: scenarios must be separate from the instructions. If Claude reads the test cases before generating the output, it optimizes for passing the tests rather than doing good work. It's the same logic as holdout data in machine learning — if the model trains on the test set, the test is meaningless.

Prompt #2 — Write verification scenarios
I need verification scenarios for this task: [SAME TASK FROM PROMPT #1]. Create a _scenarios-[task-name].md file with 3-5 scenarios. Each scenario should have: 1. **Scenario name** — a short descriptive title 2. **Input conditions** — what specific situation or input triggers this scenario 3. **Expected behavior** — what the output should do in this situation 4. **Observable criteria** — specific, checkable things I can verify (not "it should be good" but "it should include X, Y, and Z") Include 1-2 **negative scenarios** — things the output should NOT do. These catch common failure modes: being too generic, hallucinating details, wrong tone, missing key information. IMPORTANT: Save this file separately from any instructions or specs. When I run the task, Claude should NOT read the scenarios file. Scenarios are for evaluation only — they're the holdout set. Format each scenario clearly so another person (or Claude in a different session) could use them to evaluate output without needing additional context.
Keep them separate. Always.
If Claude sees the scenarios before generating output, the evaluation becomes meaningless. Claude will pattern-match to the test cases instead of doing genuine work. The discipline is simple: scenarios live in a separate file. The session that generates output never reads that file. A different session evaluates the output against the scenarios.

StrongDM's Software Factory uses this exact pattern at scale. They call it "satisfaction testing" — scenarios stored separately from the spec, evaluated by an independent judge. The principle comes from machine learning: a model evaluated on its training data tells you nothing about real performance. The same applies to AI output. If the system that generates the output also designed the test, the test is compromised.

When you evaluate AI output, most people only look at one thing: is the output itself good? But that's only one of three lenses you should be using. The other two are where the real improvement lives.

Lens 1
Output Quality
Does the artifact itself meet quality standards? Grammar, format, completeness, correctness.
Lens 2
Goal Achievement
Does it accomplish the intent? Persuasion, accuracy, actionability — whatever the work was for.
Lens 3
System Diagnosis
If it fails, where in the system did it break? Data gap? Instruction gap? Wrong model?

Most people only use Lens 1. They read the output, decide if it's "good enough," and either use it or regenerate. That's a coin flip, not a system.

Lens 2 asks the intent question: did this output actually accomplish what I needed it to? A perfectly formatted report that surfaces the wrong insights fails Lens 2. A well-written email that doesn't get the team aligned fails Lens 2. Quality and achievement are different things.

Lens 3 is where improvement lives. When output fails — and it will — the question isn't "how do I fix this output?" It's "where in my system did this break?" Was the data layer missing critical context? Were the instructions ambiguous? Was the quality standard incomplete? Lens 3 turns failures into system improvements.

Bad output isn't a failure. Bad output you can't diagnose is a failure.

When you can trace a problem to its source — "this failed because the instructions didn't specify audience" or "this failed because the data layer is missing Q4 numbers" — the system gets better. When you just say "this isn't good, try again," nothing improves.

You already know Lens 3
In the execution layer tutorial, you asked Claude "where did you have to guess?" That diagnostic question is Lens 3 applied during execution. Now you're applying it after execution — using structured scenarios and a separate evaluator instead of relying on in-session self-reporting.

Time to put it all together. You're going to run a real task, then evaluate the output against your scenarios in a completely separate session. This separation is the core discipline of verification.

Prompt #3 — Execute the task
Run [YOUR TASK] using the instructions in [YOUR PLAYBOOK/SKILL/INSTRUCTIONS FILE]. Use the data layer via _context-index.md. Follow the instructions exactly — don't ask me clarifying questions, just work from what you have. Save the output as _output-[task-name].md when you're done.
Don't evaluate in the same session
Once Claude generates the output, close this session or start a new one. The session that builds should never judge. If the same session evaluates its own work, it's grading its own homework — it will be biased toward justifying its choices rather than objectively evaluating the result.

Now, in a fresh session, run the evaluation.

Prompt #4 — Evaluate against scenarios
I need you to evaluate an output against a set of verification scenarios. Read these two files: 1. The output: _output-[task-name].md 2. The scenarios: _scenarios-[task-name].md For each scenario: 1. **Scenario name** — which scenario you're evaluating 2. **Verdict: YES or NO** — does the output satisfy this scenario? 3. **Evidence** — quote or reference the specific part of the output that passes or fails. Be concrete. After evaluating all scenarios, give an overall verdict: - **PASS** — all scenarios satisfied - **PARTIAL** — some scenarios satisfied, some failed - **FAIL** — most scenarios failed End with a **diagnostic summary**: For any failed scenarios, identify the most likely root cause. Was it a data gap (missing information), an instruction gap (unclear or incomplete instructions), a quality gap (the work was done but not well enough), or a scenario gap (the scenario itself was unrealistic or poorly defined)? Save this evaluation as _eval-[task-name].md.
This is the LLM-as-judge pattern
You're using one Claude session to evaluate another's work. This isn't perfect — AI judges have biases too — but it's dramatically better than self-evaluation. The separation of concerns (builder vs. judge) combined with concrete scenarios produces evaluations you can actually act on. Over time, you calibrate the scenarios based on whether the evaluations match your own judgment.

You have an evaluation. Some scenarios passed. Some probably didn't. The instinct is to fix the output — tweak the wording, add the missing section, adjust the format. That instinct is wrong.

Fixing the output is a one-time patch. It makes this output better but leaves the system unchanged. The next time you run the task, the same failures will recur. The high-leverage move is to trace each failure back to its source and fix the system instead.

Prompt #5 — Diagnose and fix the system
I ran a task and evaluated it against my scenarios. Some scenarios failed. I want to fix the system — not just the output. Read the evaluation: _eval-[task-name].md For each failed scenario: 1. **Which lens failed?** - Lens 1 (Output Quality) — the artifact itself has quality issues - Lens 2 (Goal Achievement) — the output doesn't accomplish the intent - Lens 3 (System Diagnosis) — trace to root cause 2. **Root cause diagnosis:** - **Data gap** — Was critical information missing from the data layer? What file or knowledge should be added? - **Instruction gap** — Were the instructions ambiguous or incomplete? What specific instruction should be added or changed? - **Quality standard gap** — Is there a quality dimension not captured in _quality-standard.md? - **Scenario gap** — Is the scenario itself poorly defined or unrealistic? Should it be rewritten or removed? 3. **Proposed fix** — The specific change to make. Not "improve the instructions" but exactly what to add, edit, or create in which file. Organize the fixes by layer: - **Data layer fixes** — files to add or update - **Execution layer fixes** — instruction changes - **Verification layer fixes** — scenario or quality standard updates Don't propose changes to the output itself. Only propose changes to the system that produces the output.
You're not fixing outputs. You're improving the machine that makes outputs.
Every fix you apply to the system improves every future run — not just this one. Data layer fixes mean richer context for all tasks. Instruction fixes mean fewer guesses across all executions. Quality standard fixes mean better evaluations everywhere. The system compounds.

Review Claude's proposed fixes. Apply the ones that make sense. Some fixes will be to the data layer (add a file, enrich an existing one). Some will be to the execution layer (clarify an instruction, add a guardrail). Some will be to the verification layer itself (sharpen a scenario, add a new one you hadn't thought of).

The failures are the most valuable part.

A task that passes all scenarios tells you the system works. A task that fails tells you exactly where to improve it. Failed runs produce system improvements. Successful runs just confirm what you already built. Welcome the failures.

Verification isn't a one-time exercise. It's a practice — a habit you build into how you work with Claude. The question is how deep to go for different kinds of work.

Three levels of verification practice
Spot check
Run scenarios manually after important outputs. Quick sanity check. 5 minutes.
Pre-flight
Write scenarios before every significant task. Define success first, then execute. 15 minutes.
Systematic
Build scenario libraries by domain. Reuse across tasks. Every run gets evaluated. The full system.

Start at level 1. Spot-check your most important outputs against the scenarios you just wrote. As the habit builds, move to pre-flight: writing scenarios before every significant task. Over time, you'll build scenario libraries that you reuse across similar tasks.

One-off creative tasks don't need formal scenarios. If you're brainstorming ideas, drafting a one-time message, or exploring a new concept, verification adds friction without value. The question to ask is: will I do this again?

If the answer is yes — if this is a recurring task that produces work others rely on — then verification is an investment that pays off every future run. If it's truly one-time, your own judgment is sufficient.

Start with 3-5 scenarios per task. That's enough to catch the most common failure modes without making evaluation burdensome. As you discover new ways the output can fail — and you will — add scenarios for those specific failures.

A mature task might have 10-15 scenarios, but you build up to that. Each scenario should earn its place by catching a failure that actually matters. If a scenario has never failed and never would, remove it — it's adding noise, not signal.

Calibrate your scenarios against your own judgment
After the evaluator (Claude in a fresh session) grades the output, check whether you agree. If the evaluator passes something you'd fail, the scenario is too loose. If it fails something you'd pass, the scenario is too strict or poorly defined. The scenarios should converge with your judgment over time — they're encoding your taste into a repeatable process.
