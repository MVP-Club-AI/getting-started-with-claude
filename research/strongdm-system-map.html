The Pipeline
Human Intent
Specifications, Scenarios, and Seeds
Humans define what the system should do — never how. Four input types seed everything downstream. This is the only place humans directly author content.
Input: Specs Input: Scenarios Input: Screenshots Input: Exemplars Leverage: Spec precision Leverage: Scenario quality
Specifications (NLSpecs)
Natural-language documents describing desired behavior, constraints, and architecture. Written by humans; implemented by agents. The spec is the product — code is just the compilation artifact.
Scenarios (Holdout Set)
End-to-end user stories stored outside the codebase — like a holdout set in ML. Prevents agents from gaming tests. Evaluated by LLM-as-judge, not boolean assertions.
Screenshots & Mockups
Visual references that seed frontend development. Agents interpret visual intent directly from images — another input channel that bypasses the need for human code.
Exemplars (Gene Transfusion)
Working code from other codebases that demonstrates a pattern. Agents extract the pattern and synthesize it into the target codebase. Example: using Caddy's Let's Encrypt integration as reference.
"If what you described was ambiguous, you're going to get software that fills in the gaps with software guesses, not customer-centric guesses."

Orchestration
Attractor — Pipeline Engine
A DOT/Graphviz-based pipeline orchestrator. Nodes are tasks, edges are transitions. The open-source repo contains zero code — only three NLSpec files that any agent can implement. 11+ community implementations exist.
Decision: Conditional branching Decision: Human gates Leverage: Model stylesheets
Node Types (Graphviz shapes → handlers)
box → codergen (LLM task) · hexagon → wait.human (approval gate) · diamond → conditional (branching) · component → parallel (concurrent branches) · parallelogram → tool (external command) · house → stack.manager_loop (recursive supervision)
Model Stylesheets
CSS-like syntax for per-node LLM configuration. Example: .critical { llm_model: claude-opus; reasoning_effort: high; } — different models and settings for different tasks in the same pipeline.
Crash Recovery
Checkpoint-based state snapshots at every node. Pipeline can crash and resume from exactly where it left off. Serializable state enables distributed execution.
Five-Phase Execution
PARSE → VALIDATE → INITIALIZE → EXECUTE → FINALIZE. The engine traverses the graph from start to exit, with exponential backoff retries (up to 50), goal gate enforcement, and conditional edge selection using boolean expressions.

Execution
Coding Agent Loop
The inner loop: LLM calls paired with tool execution. Provider-native toolsets (no universal shim — each model gets its native format). Loop detection prevents agents from spinning. Mid-session steering allows course correction without restarting.
Leverage: Model selection Decision: Loop detection Decision: Steering injection Output: Code changes
Provider-Native Toolsets
OpenAI agents get apply_patch. Anthropic agents get old_string/new_string. Each provider's native tool format, not a forced universal abstraction. Captures provider-specific capabilities like reasoning tokens and extended thinking.
Session State Machine
IDLE → PROCESSING → (AWAITING_INPUT | IDLE | CLOSED). Tool execution environment abstraction supports local, Docker, Kubernetes, WASM, or SSH backends. Output truncated to 8KB for the LLM; full output preserved in logs.
Loop Detection
Tracks tool call signatures over a sliding window (default 10 calls). Detects repeating patterns and breaks cycles — preventing agents from getting stuck hammering the same failing approach.
Mid-Session Steering
steer() injects corrections into an active session. follow_up() adds new inputs. Subagents get independent conversation histories but shared filesystem access.

Validation Environment
Digital Twin Universe (DTU)
Behavioral clones of third-party SaaS services — standalone Go binaries that replicate APIs, edge cases, and observable behaviors. Agents test at unlimited volume without rate limits, API costs, or abuse detection.
Leverage: Clone fidelity Leverage: Scenario coverage
Current Behavioral Clones
Okta (identity/auth) · Jira (issue tracking) · Slack (messaging) · Google Docs, Drive, Sheets (productivity). Each clone targets 100% compatibility with popular SDK client libraries.
  • Validate at volumes exceeding production rate limits
  • Test failure modes that would be dangerous against live services
  • Thousands of scenarios per hour — no API costs
  • Deterministic, replayable test conditions
  • Requires substantial engineering investment to build each clone
▾ Details
Evaluation
Satisfaction Testing
Not pass/fail. An LLM-as-judge evaluates: "Of all observed trajectories through all scenarios, what fraction likely satisfies the user?" Probabilistic and empirical — mirrors ML evaluation methodology, not traditional QA.
Decision: Satisfaction threshold Decision: Model consensus Output: Satisfaction score
"Satisfaction" replaces "correctness." A scenario can be satisfied in multiple ways. The LLM judge evaluates behavioral outcomes against user intent — not line-by-line code inspection.
Why Not Boolean Tests?
They observed agents writing return true to pass self-written tests while doing nothing useful. Scenarios stored outside the codebase prevent this gaming. The holdout set can't be overfit.
The ML Analogy
Generated code = model weights. Scenarios = holdout evaluation set. Satisfaction score = model accuracy. The code is an opaque artifact — you validate behavior, not implementation.

Supporting Infrastructure
CXDB
AI context store. Turn DAG + content-addressed blob storage. O(1) conversation forking, 70%+ storage reduction. Every turn of every conversation, preserved and branchable.
Leash
Container-based agent security harness. Cedar policy language, full telemetry, MCP tool call inspection. Wraps any agent (Claude, Codex, Gemini) in an auditable sandbox.
Weather Report
Public model-selection matrix. Which model for which task — including consensus voting (multiple models for sprint planning). Updated as model capabilities shift.
Key Techniques
Shift Work
Separate interactive specification from non-interactive execution. Humans spec first; agents execute without interruption.
Leverage
Gene Transfusion
Move working code patterns between codebases by pointing agents at exemplars. Identify → Extract → Synthesize → Validate → Propagate.
Input technique
Semport
Automated semantic porting between languages. StrongDM monitors OpenAI's Python SDK daily and auto-translates patterns into Go.
Leverage
Pyramid Summaries
Multi-zoom-level summarization (like map tiles). Survey hundreds of items at 2-word detail, expand only what's interesting. MapReduce for context.

The Filesystem

Open Questions and Critical Concerns
Stanford Law
The circularity problem: "When the builder and the inspector share the same blind spots, no amount of test variety fully eliminates risk." Who's liable when agent-written, agent-tested code fails?
Simon Willison
Economics: $1K/day per engineer = ~$20K/month token budgets. Also: if agents can build features in hours, competitors can clone your product just as fast. Defensibility shifts from code to data and specifications.
Ry Walker
Applicability: DTU approach is optimized for integration-heavy software (StrongDM's core business). Building behavioral clones requires substantial engineering investment. Unclear how well this generalizes to other domains.
Structural
Spec quality bottleneck: The system's ceiling is the human's ability to write precise specifications and comprehensive scenarios. Garbage specs in, garbage code out — at scale and speed.
